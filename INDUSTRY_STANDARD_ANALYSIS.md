## 4. in-toto Framework

### Source: [in-toto/docs v1.0](https://github.com/in-toto/docs/blob/v1.0/in-toto-spec.md)

in-toto requires three distinct metadata types:

| in-toto Artifact | Purpose | MKPE Evidence | Verdict |
|------------------|---------|---------------|---------|
| **Root layout** | Defines supply-chain steps, their ordering, expected materials/products, and authorized functionaries | Not implemented | **Fail** |
| **Link metadata** | Records what files entered and exited each step | Not implemented | **Fail** |
| **Step rules** | Pattern matching (CREATE, MODIFY, DELETE, ALLOW, DISALLOW) to detect unexpected files | Not implemented | **Fail** |

MKPE has **no concept of supply-chain topology**. It attests "this file was signed by this key at this time," but it cannot express "Step A must run before Step B, and Step B's output must be a superset of Step A's output."

**in-toto Assessment:** Major architectural gap. MKPE is a signing tool; in-toto is a supply-chain integrity framework. The two are complementary but not interchangeable.

## 5. TUF (The Update Framework)

### Source: [theupdateframework.io/specification v1.0.34](https://theupdateframework.github.io/specification/latest/)

TUF's core security model is **role-based delegation with threshold signatures and consistent snapshots**.

| TUF Requirement | Spec says | MKPE Evidence | Verdict |
|-----------------|-----------|---------------|---------|
| **Root metadata** | Ships with trusted root keys; delegates to targets/snapshot/timestamp | No root metadata format; no POUF | **Fail** |
| **Targets role** | Signs metadata describing target files | `Manifest` signs bundle root hash, but not in TUF metadata format | **Fail** |
| **Snapshot role** | Prevents mix-and-match by signing latest version of all targets metadata | Not implemented | **Fail** |
| **Timestamp role** | Prevents indefinite freeze with short-lived timestamp metadata | Not implemented | **Fail** |
| **Threshold / quorum** | Role requires N-of-M signatures | `MultiSignatureManifest` (`multisig.rs:42`) supports threshold for *manifests*, but not for *roles* | **Partial** (concept present) |

## 7. Merkle Audit Trail

### What MKPE implements

| Feature | Implementation | Evidence |
|---------|---------------|----------|
| Chained audit events | `AuditEvent::previous_hash` links to prior event hash | `core/src/audit.rs:186-189` |
| Chain verification | `AuditLog::verify_chain()` recomputes hashes and checks linkage | `core/src/audit.rs:166-178` |
| Merkle root | `AuditLog::compute_merkle_root()` builds a tree over all entries | `core/src/audit.rs:156-163` |

### What industry standards require

A real transparency log (e.g., Rekor, Certificate Transparency, Trillian) adds:

1. **Append-only property with cryptographic proof** — any fork of the log is detectable.
2. **External witnesses / monitors** — independent organizations continuously verify the log.
3. **Signed tree head (STH)** — a periodically published signed root that clients can check against.
4. **Inclusion proof / consistency proof** — clients can prove a specific entry is in the log, and that the log hasn't been rewritten.

MKPE has none of these. The Merkle root is computed on demand from a local file. An attacker with filesystem access can rewrite the log and recompute a valid root.

**Merkle Assessment:** Good local integrity primitive; not a transparency log.

## 8. Policy Engine

### What MKPE implements

- `PolicyCondition` enum: `KeyVersion`, `TimestampRange`, `KeyId`, `ChainDepth`, `BundleHash`, `NonceFresh`
- JSON-configurable policy files
- AND/OR evaluation over conditions

### What real policy engines do (e.g., OPA/Rego, in-toto layouts)

- **OPA/Rego:** Turing-complete policy language; can express arbitrary rules over complex JSON structures.
- **in-toto layouts:** Define step ordering, artifact allow-lists, and functionary authorization in a single signed root document.

MKPE's policy engine is a **static manifest field checker**. It cannot express "only artifacts built by GitHub Actions runners with repository X are allowed."

## 9. Replay Protection

| Feature | MKPE Implementation | Strength |
|---------|--------------------|----------|
| `nonce` field in `Manifest` | `Option<u64>` set to nanosecond timestamp | Weak: nanosecond clock is monotonic but not cryptographically random; predictable range |
| `nonce` field in `BuildAttestation` | Same pattern | Weak |
| `verify_nonce(min)` | Checks `nonce >= min` | Weak: no server-provided challenge or freshness token |

**Industry-standard replay protection** uses:
- Server-issued nonces (challenge-response)
- TLS session binding
- Short-lived timestamps with trusted time source (RFC 3161)

MKPE's replay protection is **local-clock-based best effort**. It prevents accidental replay by well-behaved local tools, not malicious attackers.

## 11. What "Industry +3" Actually Means

The user stated the goal was "one level higher than the current industry +3 standard."

### If "industry +3" = SLSA Build Level 3:
MKPE is **not Level 3**. It lacks:
- Hosted build platform
- Isolated/ephemeral build environment
- Build platform control plane that keeps signing keys secret

### If "industry +3" = "Architecture Layer 3" (MKPE internal model):
The `ARCHITECTURE_LAYERS.md` document describes:
- Layer 1: Content addressing
- Layer 2: Cryptographic proofs
- Layer 3: Attestations
- Layer 4: (planned) Policy / transparency

MKPE *has* implemented Layer 3 (attestations) and some Layer 4 surface area (policy engine, Merkle audit). In this **internal model**, MKPE is arguably at "Layer 3 with early Layer 4 primitives."

**Honest translation:** Relative to *actual* industry standards (SLSA, in-toto, Sigstore, TUF), MKPE is at:

> **SLSA Build Level 0–1** (provenance exists and is signed, but no build platform guarantees)  
> **DSSE v1.0 compliant** (envelope format is correct)  
> **Not in-toto, TUF, or Sigstore conformant** (missing entire trust architectures)

## 13. Recommendations to Close Gaps

1. **If targeting SLSA Build L2:** Integrate with a CI/CD runner (GitHub Actions, GitLab CI) so provenance is generated by the runner, not the user's laptop. Generate the signed attestation inside a runner step, and expose the runner's identity in the attestation.

2. **If targeting SLSA Build L3:** Run builds inside an ephemeral container or VM where the signing key is injected by the platform (not stored in the repo or on disk). Use HSM or KMS for signing.

3. **If targeting Sigstore equivalence:** Publish signatures to Rekor using the `hashedrekord` entry type. This requires HTTP client integration and hash-based entry creation.

4. **If targeting TUF equivalence:** Implement `root.json`, `targets.json`, `snapshot.json`, and `timestamp.json` metadata formats with version numbers and expiration dates. Implement the TUF client update workflow.

5. **If targeting in-toto equivalence:** Define a `layout` schema, record `link` metadata per build step, and implement rule evaluation (CREATE, MODIFY, DELETE, ALLOW, DISALLOW).
